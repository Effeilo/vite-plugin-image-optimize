# Hooks

`vite-plugin-image-optimize` implements three Vite/Rollup plugin hooks. This page documents what each hook does and when it runs.

---

## Overview

| Hook | Phase | Type | Purpose |
|---|---|---|---|
| `configResolved` | Config | Sync | Resolve `publicImgDir` and `distImgDir` from Vite config |
| `buildStart` | Pre-build | Async | Generate `.webp` and `.avif` from JPEG/PNG in `public/img` |
| `closeBundle` | Post-build | Async | Compress all images in `dist/img` in-place |

The plugin also declares `apply: 'build'`, which prevents all hooks from registering during `vite dev` or `vite preview`.

---

## `configResolved`

**Type:** Sync  
**Runs:** Once, after Vite resolves and merges all configuration.

```ts
configResolved(config: ResolvedConfig) {
  publicImgDir = path.join(config.publicDir, 'img')
  distImgDir = path.resolve(config.root, config.build.outDir, 'img')
}
```

Reads `config.publicDir` and `config.build.outDir` from Vite's resolved configuration and stores them as the source and target directories for subsequent hooks.

This approach is more reliable than using `process.cwd()` directly: it respects any custom `publicDir` or `build.outDir` values defined in the project's Vite config.

---

## `buildStart`

**Type:** Async  
**Runs:** Once, before Vite begins processing modules.

Generates `.webp` and/or `.avif` versions of JPEG and PNG files found in `<publicDir>/img/`.

### Behavior

1. If both `webpEnable` and `avifEnable` are `false`, the hook returns immediately.
2. If `<publicDir>/img/` does not exist, the hook returns silently.
3. All JPEG and PNG files are collected via `getAllImages`.
4. For each source file, WebP and/or AVIF versions are generated using `sharp`.
5. Each output file is checked against `fileIsNewer` - if the output already exists and is newer than the source, the conversion is skipped.
6. All conversions run in parallel with `Promise.all`.
7. The results are printed to the terminal.

### Output location

Generated files are written alongside the source files in `<publicDir>/img/`. Vite then copies the entire `public/` directory to `dist/` during its own asset pipeline.

```
public/img/
├── logo.png           (source)
├── logo.webp          (generated by buildStart)
├── logo.avif          (generated by buildStart)
```

---

## `closeBundle`

**Type:** Async  
**Runs:** Once, after Vite has written all output files to disk.

Compresses all images in `<outDir>/img/` in-place using [sharp](https://sharp.pixelplumbing.com/) and [svgo](https://svgo.dev/).

### Behavior

1. If `<outDir>/img/` does not exist, the hook returns silently.
2. All image files are collected via `getAllImages` (`.jpg`, `.jpeg`, `.png`, `.svg`, `.webp`, `.avif`).
3. For each file:
   - The file content is read into memory using `fs.readFile`.
   - The buffer is passed to `sharp(buffer)` (not `sharp(filePath)`) to avoid file locking on Windows.
   - The compressed output is written back to the same path using `fs.writeFile`.
4. SVG files are optimized using `svgoOptimize` with `multipass: true`.
5. All compressions run in parallel with `Promise.all`.
6. A per-file report and total savings summary are printed to the terminal.

### Why `sharp(buffer)` instead of `sharp(filePath)`

On Windows, passing a file path to `sharp()` keeps a native file handle open until the sharp instance is garbage-collected. When `fs.writeFile` then tries to open the same file for writing, it encounters a locked file and throws an `UNKNOWN` error. Using the pre-loaded buffer avoids this entirely.

### Per-format settings

| Extension | Encoder | Settings |
|---|---|---|
| `.jpg` / `.jpeg` | sharp JPEG | `quality`, `mozjpeg: true` |
| `.png` | sharp PNG | `quality`, `compressionLevel: 9` |
| `.webp` | sharp WebP | `quality` |
| `.avif` | sharp AVIF | `quality` |
| `.svg` | svgo | `multipass: true` |
